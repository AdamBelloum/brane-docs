6. Virtual Machine (VM) Design

The Brane Virtual Machine (VM) executes workflows by interpreting WIR graphs.

The VM:

  • Maintains runtime state (stacks, registers, frames),
  • Follows edges in WIR graphs,
  • Executes edge instructions that manipulate values and control flow.

Frontend languages do not interact directly with the VM; they compile into WIR.

6.1. Execution model and invariants

Execution steps:

  1. Load WIR program:
  2. Main graph,
  3. Function graphs,
  4. Symbol table,

  5. Results map.

  6. Initialize state:

  7. Clear expression stack,
  8. Initialize variable register,

  9. Push initial frame for main graph.

  10. Execute edges:

  11. Start at main entry edge,

  12. For each edge:

    • Run edge instructions,
    • Move program counter according to edge kind.

  13. Terminate:

  14. On a Stop edge or when frames and edges are exhausted.

Core invariants:

  • Type safety – VM assumes WIR is type-checked.
  • Frame isolation – each function call has its own frame and variable scope.
  • Deterministic edge execution – given WIR and initial state, internal behaviour is deterministic (external calls may introduce non-determinism).

6.2. Core runtime structures

The VM uses:

  • Expression stack,
  • Variable register,
  • Frame stack.

6.2.1. Expression stack

The expression stack holds temporary values used in evaluation:

  • Frontend expressions compile to:
  • Push operands,
  • Apply operations,
  • Pop or leave results on the stack.

Operations include:

  • Arithmetic: Add, Sub, Mul, Div, Mod.
  • Comparisons: Eq, Ne, Lt, Le, Gt, Ge.
  • Boolean logic operations where defined.

Arrays:

  • Array – build arrays from stack values.
  • ArrayIndex – retrieve elements by index.

Values on the stack are typed; instructions operate on specific types.

6.2.2. Variable register

The variable register stores named variables and their values:

  • Variables are declared with types in WIR.
  • Types do not change after declaration.
  • VM uses variable IDs (from symbol table) to load/store values.

Typical instructions:

  • LoadVar – push variable value onto stack.
  • StoreVar – pop stack value into variable.

6.2.3. Frame stack

The frame stack manages function calls and scopes. Each frame holds:

  • The current graph (main or function),
  • The current program counter,
  • Variable bindings in that call.

On Call edges:

  • VM:
  • Pushes a new frame,
  • Sets graph to the callee’s function graph,
  • Transfers arguments to parameter variables.

On Return edges:

  • VM:
  • Places return values on stack or in variables,
  • Pops the frame,
  • Restores caller’s graph and program counter.

This supports nested/recursive calls and proper scoping.

6.3. Edge instructions

Each edge contains a sequence of edge instructions, which are atomic operations for the VM.

Instruction categories:

6.3.1. Arithmetic and logic

  • Add, Sub, Mul, Div, Mod
  • Eq, Ne, Lt, Le, Gt, Ge
  • Boolean operations (e.g., And, Or, Not) where defined.

All operate on stack values and push results.

6.3.2. Data structure operations

  • Array – construct arrays from stack values.
  • ArrayIndex – read array element at an index.

6.3.3. Variable operations

  • LoadVar – read variable value, push onto stack.
  • StoreVar – pop stack value, store in variable.

6.3.4. Control and call support

Edge instructions may:

  • Prepare values used by branches and loops (conditions, counters).
  • Prepare arguments for Call edges.
  • Place return values for Return edges.

Core call/return semantics are encoded in edge kinds; instructions support data preparation.

6.3.5. External calls and plugins

Some instructions trigger external behaviour:

  • Plugin calls – for operations not purely internal.
  • External ECU calls – delegating package execution to workers.

These integrate with:

  • Planner (placement),
  • Workers (execution),
  • Results map (recording outputs and locations).

6.4. Execution example (high level)

Conceptual execution of a workflow:

  1. Initialize main frame and program counter.
  2. Execute edges in main:
  3. Update stack and variables.

  4. Move to next edges based on edge kinds.

  5. On Call edge:

  6. Evaluate arguments.
  7. Push callee frame.
  8. Transfer arguments to callee parameters.

  9. Jump to callee entry edge.

  10. In callee:

  11. Execute edges and possibly call other functions or packages.

  12. Reach Return edge.

  13. On return:

  14. Place return values for caller.
  15. Pop callee frame.

  16. Resume caller’s control.

  17. Intermediate results:

  18. Produced by tasks and external calls.
  19. Recorded in results map with locations and access info.

  20. Consumed by later edges and tasks.

  21. Termination:

  22. Main graph reaches Stop edge.
  23. Workflow completes; final results are available as datasets or values.

6.5. Summary

The Brane VM is a stack-based interpreter for WIR graphs with:

  • Expression stack for temporary values,
  • Variable register for named, typed variables,
  • Frame stack for function calls and scopes,
  • Edge instructions implementing arithmetic, logic, data structures, variables, control support, and external calls.

It enforces WIR semantics and integrates with domain infrastructure and workers to execute federated workflows predictably and safely.
